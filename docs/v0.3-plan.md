# seeklink v0.3 plan (v2, post-review)

Two changes. All local to seeklink. No new Python dependencies. No model changes.
Targeting `0.3.0` as a single release.

> **v2 changes from v1.** Consensus bonus dropped (formula was miscopied from
> qmd; math uncalibrated; no data to justify). Line-range scope expanded to
> cover the daemon and CLI output layers that v1 missed. Baseline definitions
> tightened after `__main__.py` cold-start path was found to differ from the
> daemon path.

## Scope

| # | Item | Why now | Risk |
|---|------|---------|------|
| 1 | Position-aware rerank blending | Fixes known failure mode: reranker overrides confident first-stage wins | Medium |
| 2 | Line-range retrieval (full pipeline: SearchResult + daemon + CLI + `get`) | Unblocks agent workflows on long log files | Low–Medium |

Explicitly **not** in v0.3:

- Query expansion (v0.4, gated on `docs/blind-test.md` pass criteria)
- Embedder upgrade (v0.5+, only if blind test shows vector channel is the weak link)
- RRF consensus / top-rank bonus — **dropped from v0.3**. Codex review
  confirmed qmd's `topRank = min(rank across channels)` formula does not map
  to "multiple channels rank 1"; the `+0.05` / `+0.02` constants are
  uncalibrated against seeklink's 1.0 / 1.0 / 0.3 / 1.5 channel weights;
  rank-1 vs rank-2 RRF delta for unit-weight channels is only `1/61 - 1/62
  ≈ 0.000264`, so any bonus >= 0.01 would dwarf the fusion signal. Revisit
  with our own labeled data.
- Hierarchical context (`context add`) — deferred (not "never"; qmd does use
  context as a retrieval/rerank signal at `qmd/src/store.ts:3298,4192,4266`;
  v0.3 skips it because we have no UI consumer yet)
- `multi-get` by glob, AST-aware chunking, MCP resurrection — no demand
- RRF `k` constant stays at 60

## Known seeklink inconsistency to fix alongside (not a new feature, just parity)

Cold-start search path (`seeklink/__main__.py:229-236`) calls `search(db,
embedder, query, **search_kwargs)` without constructing or passing a
`Reranker`. Daemon path (`seeklink/daemon.py:188-200`) passes
`reranker=reranker` by default. Result: the same `seeklink search` command
silently produces different rankings depending on whether the daemon is up.

**Fix** (bundled with Item 1 since both touch the reranker path): construct
a `Reranker()` in the cold-start branch and pass it to `seeklink_search`.
Matches daemon behavior. Same env-var opt-out (`SEEKLINK_RERANKER_MODEL=""`)
still disables it via the `Reranker.disabled` flag. 3-5 lines in
`__main__.py`.

## Item 1: Position-aware rerank blending

### Problem
Current reranker **fully replaces** RRF scores
([seeklink/search.py:219-234](../seeklink/search.py)):

```python
results = [SearchResult(..., score=float(rerank_scores[i])) for i, r in ...]
results.sort(key=lambda r: r.score, reverse=True)
```

Failure mode: when you search `[[fsrs-basics]]`, the title channel pushes
the exact-title doc to RRF rank 1. The reranker reads that doc's content,
decides a longer adjacent doc "reads more relevant", and demotes the exact
alias hit. We lose a confident first-stage win to reranker's
content-blindness.

### Change
Blend a position-based retrieval signal with the reranker score, weighted
by RRF rank. Copied verbatim from qmd
([src/store.ts:4230-4234](https://github.com/tobi/qmd/blob/main/src/store.ts)):

```python
for i, r in enumerate(results):
    rrf_rank = i + 1                    # 1-indexed rank after RRF sort
    rank_pos_score = 1.0 / rrf_rank      # 0-1 range, aligns with rerank scale
    if rrf_rank <= 3:    alpha = 0.75
    elif rrf_rank <= 10: alpha = 0.60
    else:                alpha = 0.40
    blended = alpha * rank_pos_score + (1 - alpha) * rerank_scores[i]
    # build new SearchResult with score=blended
```

### Why copy qmd verbatim (and not design our own)
- `1/rank` and reranker sigmoid-prob are both 0-1 → no scale mismatch.
  seeklink's current RRF score is `~1/(60+rank) ≈ 0.016–0.03` — blending it
  with a 0-1 rerank score without normalization would drown the RRF signal.
- The 0.75/0.60/0.40 weights are empirical in qmd but used in production.
  Without our own labeled data, any weights we pick are vibes.
- Trivially reversible: one function, ~12 lines.

### Files
- `seeklink/search.py`: replace lines 219-236 (rerank branch). Keep the
  "reranker disabled → fall through to RRF order" path untouched.
- `seeklink/__main__.py`: construct `Reranker()` in `_cmd_search` cold-start
  branch (fix parity with daemon, see top section).
- `tests/test_search.py`: fixture where title channel ranks doc A at #1 and
  content channel ranks doc B at #1 with a higher rerank score; assert
  doc A survives at rank 1 after blending (expected because
  `0.75 * 1.0 = 0.75` dominates a low rerank on doc A).

### Risk + mitigation

- **All top_k candidates get blended, not just top 20.**
  `seeklink/search.py:170-173` sets `candidate_k = max(rerank_k=20, top_k)`.
  If the caller passes `--top-k 30`, positions 21–30 get `alpha=0.40` —
  reranker dominates those, which is exactly the qmd intent but worth
  documenting.
  - *Mitigation*: test `top_k=30` explicitly; document in CHANGELOG.

- **Rerank score distribution is sharp (near 0 or near 1); rank signal is
  smooth**. Mixing can create close blended scores that tie.
  - *Mitigation*: debug mode (`SEEKLINK_DEBUG=1`) logs
    `(rrf_rank, rank_pos_score, rerank_score, blended)` per result.
    Eyeball the first 10 real queries on rhizome vault before merging.

- **Reranker-disabled path must stay correct** (Linux without MLX, explicit
  opt-out). Existing `if reranking_enabled` branch untouched — blending only
  activates when reranker runs.

## Item 2: Line-range retrieval (full pipeline)

### Problem
Agent using seeklink today:

1. `seeklink search "FSRS 实现细节"` → chunk with ~400-token snippet, no idea
   where in the file.
2. Agent wants 50 lines of context around the match → issues `Read` on the
   full file. For `logs/rhizome-dev/2026-W15.md` that's ~8k tokens to get
   maybe 300 used. Wasted context window.
3. No way to ask "give me lines 120-180 of that file" without first reading
   the whole thing.

### Change, 2a — `SearchResult` gains line numbers

Add `line_start: int` and `line_end: int` (both 1-indexed, inclusive) to
`SearchResult`, computed from the backing `Chunk`'s
`char_start / char_end` **against the current on-disk file content**.

**Chunk span semantics (important)**: `ChunkSpan.char_end` is
*end-exclusive* — the invariant in [chunker.py:3](../seeklink/chunker.py)
says `chunk.text == source_text[chunk.char_start:chunk.char_end]`. To get
an **inclusive** last-character offset for line mapping, use
`max(char_start, char_end - 1)`. Using raw `char_end` over-reports
`line_end` by 1 when the chunk ends exactly on a newline.

**Data flow**: `SearchResult` today carries `chunk_id` but no `char_start`/
`char_end`. The helper that computes lines therefore needs **DB access** to
batch-load the chunks it's numbering. Signature:

```python
def compute_lines_for_results(
    db: Database,
    vault_root: Path,
    results: list[SearchResult],
) -> list[SearchResult]:
    """Return a new list of SearchResult with line_start/line_end filled in.

    Batch-fetches chunks by chunk_id (reuses db.get_chunks_by_ids, which
    already exists and is used in search.py). Caches file reads per path
    for the duration of the call.
    """
```

Because ingest strips YAML frontmatter before chunking
(`seeklink/ingest.py:70-75`), chunk offsets are **into the post-frontmatter
body, not the full file**. The mapping helper must add the frontmatter
length back:

```python
# New helper, seeklink/search.py (or a new offset_utils.py):
def body_offset_to_file_line(full_text: str, body_char_offset: int) -> int:
    """Map a char offset in the frontmatter-stripped body back to 1-indexed
    line number in the on-disk file.

    Invariant: the frontmatter prefix, if any, occupies lines 1..N of the
    file; the body starts at line N+1 in the file (where N = number of
    newlines inside the frontmatter block, inclusive of the opening/closing
    --- lines).

    If the file on disk no longer has frontmatter (user deleted it after
    indexing), treat the full file as body — that's the correct mapping
    for a body-relative offset even then.
    """
    m = _FRONTMATTER_RE.match(full_text)  # shared regex from ingest.py
    if m:
        frontmatter_len = m.end()
        prefix_lines = full_text.count("\n", 0, frontmatter_len)
    else:
        frontmatter_len = 0
        prefix_lines = 0

    body_abs_offset = frontmatter_len + body_char_offset
    lines_before = full_text.count("\n", frontmatter_len, body_abs_offset)
    return prefix_lines + lines_before + 1
```

Applied as: `line_start = body_offset_to_file_line(full_text, chunk.char_start)`
and `line_end = body_offset_to_file_line(full_text, max(chunk.char_start, chunk.char_end - 1))`.

One file read per unique `path` per search call (cache in a local dict).
`str.count("\n", 0, N)` is a C-loop, trivially fast for our file sizes.

Read file with `path.read_text(encoding="utf-8")` — Python's default
`newline=None` gives universal newline translation, so `\r\n` → `\n`
automatically. No manual CRLF handling needed.

**Edge cases** (all verified against Python stdlib behavior):

| Case | Behavior |
|---|---|
| File has no frontmatter | `_FRONTMATTER_RE.match` returns None, `prefix_lines=0`, treat full text as body |
| File moved/deleted since index | `read_text` raises `FileNotFoundError` → we log a warning, set `line_start = line_end = 0`, return result with `score` unchanged |
| File mtime newer than index `indexed_at` | Compute line numbers from current disk content (they may drift from the indexed chunk). Emit one-line stderr warning: `line numbers for {path} may drift: file modified since index`. Correctness first — stale is better than wrong. |
| Multi-byte CJK | Our char offsets are Python char (code-point) indexes, not bytes. `str.count("\n", 0, offset)` is code-point-correct. No conversion. |
| Title-only match (no chunk) | `chunk_id == 0` in existing code; set `line_start = line_end = 1` (the file itself is the match; line 1 is the least-wrong default for `seeklink get`) |
| File with BOM | `read_text(encoding="utf-8")` strips BOM if encoding is `"utf-8-sig"`, passes it through if `"utf-8"`. We use `"utf-8"` in `ingest.py:50` today; keep consistent. BOM contributes 0 newlines so no line-number impact. |

### Change, 2b — daemon response + CLI printer propagate lines

**Daemon response** (`seeklink/daemon.py:201-213`): add `line_start` and
`line_end` fields to each result dict.

```python
{
    "source_id": r.source_id,
    "path": r.path,
    "title": r.title,
    "content_preview": r.content[:200] if r.content else "",
    "score": r.score,
    "indegree": r.indegree,
    "line_start": r.line_start,   # new
    "line_end": r.line_end,       # new
}
```

**CLI printer** (`seeklink/__main__.py:179-197`): show `path:L` where L is
the starting line of the best chunk:

```python
path_with_line = f"{path}:{line_start}" if line_start > 0 else path
print(f"  {score:.4f}  {path_with_line}  {title}")
```

Title-only matches show `path:1` (the default). Path without line (stale
file error) shows `path` only with warning on stderr.

### Change, 2c — new `seeklink get` command

```
seeklink get PATH[:LINE] [-l N] [--lines N]
```

- `seeklink get notes/fsrs.md`          → full file to stdout
- `seeklink get notes/fsrs.md:120`      → 100 lines starting at 120 (default -l)
- `seeklink get notes/fsrs.md:120 -l 40` → 40 lines starting at 120
- `seeklink get notes/fsrs.md -l 50`    → first 50 lines

No DB lookup. Resolves `PATH` against vault root via `init_app(args.vault)`
(same pattern as `_cmd_search` cold-start). Exit 0 on success, exit 1 on
missing file.

**Edge cases**:

| Case | Behavior |
|---|---|
| File missing | `print(f"Error: {path} not found", file=sys.stderr); sys.exit(1)` |
| `LINE > EOF` | Empty stdout, exit 0, one-line stderr warning |
| `LINE < 1` | Treat as 1, one-line stderr warning |
| No `:LINE`, no `-l` | Print entire file (do not truncate) |
| `:LINE` without `-l` | Default 100 lines (document in `--help`) |
| File contains only `\r\n` endings | `read_text` universal-newline-translates; output is `\n`-terminated. (If the user wanted a round-trip, they would `cat` the file; this command is for agents, not backup.) |

## Performance budget

Item 1: +1 ms per query (trivial math on 20 results).
Item 2: +1 file read per *unique* path per search. For top_k=10 with
typical 5-8 unique paths, that's ~2-5 ms of file I/O on SSD. `get`
command: one file read, ~1 ms. Acceptable.

Regression target: p50 unchanged, p99 within 20% of current warm-daemon
timing (~10 ms without reranker, ~1-2 s with reranker on M3).
